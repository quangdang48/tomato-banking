## Why

Authenticated users can prove who they are (Phase 1 JWT auth) but nothing yet proves whether they are allowed to bank. Banking access (account creation, deposit, withdraw) must be gated by a verified onboarding decision. This change adds the individual eKYC onboarding slice so a customer can submit identity data for review and reach an `APPROVED` state that downstream account services can require.

## What Changes

- Add an onboarding profile per authenticated user (`INDIVIDUAL` customer type for this slice).
- Add individual KYC data capture (legal name, date of birth, identity document, Vietnam address).
- Add verification document metadata capture (metadata only; no file upload in this slice).
- Add a submit/review status workflow (`DRAFT → SUBMITTED → IN_REVIEW → APPROVED/REJECTED/REQUIRES_MORE_INFO`) with enforced transitions.
- Add an onboarding audit log row written in the same transaction as every status transition.
- Add reviewer/admin endpoints (behind a temporary admin gate) to list the review queue, start review, and decide.
- Add a centralized `requireApproved` gate that account/money-movement services call before allowing banking actions.
- Add onboarding error codes in the `2300` block and extend `GlobalExceptionHandler` for `403`, `404`, and `409`.
- Add the onboarding tables (`customer_profiles`, `kyc_verifications`, `verification_documents`, `onboarding_audit_logs`) via JPA entities, with the schema generated by Hibernate `ddl-auto` (no Flyway yet — migration tooling is adopted phase by phase later).

Out of scope (next slice / later phases): business KYB, beneficial owners, real KYC vendor integration, sanctions/PEP screening, OCR, multipart file upload/storage, and a full reviewer role model.

## Capabilities

### New Capabilities

- `kyc-onboarding`: Individual customer onboarding — profile creation, KYC data and document-metadata capture, submit/review status workflow, audit logging, reviewer decision workflow, and the `requireApproved` banking gate.

### Modified Capabilities

- None. (`auth-jwt` supplies the authenticated principal but its requirements do not change.)

## Impact

- Adds `CustomerType`, `OnboardingStatus`, `RiskLevel`, `VerificationStatus`, `DocumentStatus` enums.
- Adds entities and repositories for onboarding tables; schema is generated from the JPA entities via Hibernate `ddl-auto` (current project setting: `create-drop` on H2).
- Adds onboarding DTOs, `OnboardingService`, and `OnboardingReviewService` with implementations.
- Adds customer onboarding controller (`/api/onboarding/**`) and reviewer controller (`/api/admin/onboarding/**`).
- Adds onboarding error codes (`2300`–`2308`) to `ErrorCode` and `403/404/409` mappings to `GlobalExceptionHandler`.
- Adds `requireApproved` call sites in future account-creation and deposit/withdraw services.
- Updates `docs/design/api.md` and `docs/design/database.md` once contracts stabilize.
