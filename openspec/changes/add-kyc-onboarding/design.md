## Context

Phase 1 added stateless JWT auth (`auth-jwt`): registration, login, token validation, and a `SecurityContext` principal. The project already has `User`, `ApiResponse<T>`, centralized `BusinessException(ErrorCode)`, and `GlobalExceptionHandler`.

Auth proves identity. It does not prove eligibility to bank. Phase 2 onboarding produces one decision (`APPROVED`) that gates account creation and money movement. This change implements the individual eKYC slice only; business KYB and beneficial owners are a separate follow-up slice that extends the same profile model.

Constraints from the implementation doc and code conventions:

- No controller decides approval — that rule lives in services ([layers.md](../../../docs/code-convention/layers.md)).
- All responses use `ApiResponse<T>`; all errors flow through `BusinessException(ErrorCode)` ([exception.md](../../../docs/code-convention/exception.md)).
- Identity data is normalized and auditable; document files live outside the DB (metadata + storage reference only).
- No migration tool yet. The project runs H2 in-memory with Hibernate `ddl-auto: create-drop`; schema comes from the JPA entities. A versioned migration tool (e.g. Flyway) is adopted phase by phase later.

## Goals / Non-Goals

**Goals:**

- One onboarding profile per authenticated user, `customerType = INDIVIDUAL` in this slice.
- Capture and update individual KYC data and verification-document metadata while editable.
- Enforce the onboarding status state machine and write an audit row per transition in the same transaction.
- Reviewer workflow (start review, decide) behind a temporary admin gate.
- A single `requireApproved(userId)` policy method that account/money-movement services call.
- Onboarding error codes `2300`–`2308` and `403/404/409` handler mappings.

**Non-Goals:**

- Business KYB, beneficial owners, multiple representatives (next slice).
- Real KYC vendor integration, sanctions/PEP/watchlist screening, OCR, document authenticity.
- Multipart file upload/storage implementation (metadata only).
- A full reviewer role model — only a minimal admin gate.
- Re-verification after expiry.

## Decisions

### One profile per user, keyed by `user_id`

`customer_profiles.user_id` is `UNIQUE` and FK to `users.id`. The authenticated principal's user id resolves the profile on every customer endpoint, so customer-facing endpoints never accept a `profileId` from the client.

Alternatives considered: client-supplied profile id (rejected — lets a caller target another user's profile); many profiles per user (rejected — out of scope this phase).

### Status transitions enforced in the service layer

A central transition check rejects illegal moves with `ERROR_409_2306`. Allowed: `DRAFT→SUBMITTED`, `SUBMITTED→IN_REVIEW`, `IN_REVIEW→{APPROVED,REJECTED,REQUIRES_MORE_INFO}`, `REQUIRES_MORE_INFO→SUBMITTED`, `REJECTED→SUBMITTED`. Direct `DRAFT→APPROVED` is forbidden. Customer mutations (KYC/documents) are allowed only in `DRAFT` or `REQUIRES_MORE_INFO`.

Alternatives considered: a full state-machine library (rejected — overkill for one enum); scattering checks across controllers (rejected — violates layers.md).

### Split customer vs reviewer services

`OnboardingService` owns the customer workflow (profile, KYC, documents, submit, `requireApproved`). `OnboardingReviewService` owns the reviewer workflow (list queue, start, decide). This keeps the public surface free of admin operations and lets the admin gate evolve independently.

Alternatives considered: one service (rejected — mixes self-service and privileged operations, harder to authorize later).

### `requireApproved` is the only banking gate

Account creation and deposit/withdraw call `onboardingService.requireApproved(userId)`, which throws `ERROR_403_2302` unless status is `APPROVED`. Centralizing avoids copying status checks into controllers and keeps onboarding policy in one place.

Alternatives considered: checking status inline in each banking service (rejected — duplicated policy, drift risk).

### Audit log written in the transition transaction

Every status change writes an `onboarding_audit_logs` row (profile, actor, action, old/new status, reason) inside the same `@Transactional` boundary, so an audit gap cannot occur if the transition commits.

Alternatives considered: async/event-based audit (rejected — risks lost audit rows; out of scope).

### Schema from JPA entities, not a migration tool

The schema for the onboarding tables is declared on the JPA entities and generated by Hibernate `ddl-auto`. Constraints live in annotations: `@Table(uniqueConstraints = …, indexes = …)` for the unique keys and reviewer-queue/lookup indexes, `@Column(nullable = false, …)` for NOT NULL, and a Hibernate `@Check(constraints = "document_number ~ '^[0-9]{12}$'")` (or equivalent Bean Validation `@Pattern` on the DTO) for the CCCD format. No `V*.sql` files are added in this slice.

Alternatives considered: introduce Flyway now with `V1__auth_and_kyc.sql` (rejected per request — defer migration tooling and add it phase by phase once schemas stabilize); hand-written `schema.sql` (rejected — duplicates the entity definitions and drifts).

Trade-off: `create-drop` resets data on every restart, which is fine for this dev phase but is not production-safe. Adopting Flyway later is the planned path; entity annotations carry the constraint intent forward so the first migration baseline can be generated from them.

### Document upload is metadata-only this slice

Endpoints accept `storageKey`, filename, content type, and size — no bytes. The storage provider is chosen later; the contract is stable now.

### Privacy in responses

The customer status endpoint returns only `profileId`, `customerType`, `status`, `submittedAt`, `reviewedAt`. It MUST NOT return document numbers, full identity payloads, storage keys, or risk rationale. Full detail is reserved for reviewer endpoints (future DTOs).

## Risks / Trade-offs

- [Risk] Reviewer endpoints are exposed before a real role model exists → Mitigation: place `/api/admin/onboarding/**` behind a temporary admin gate and document it as dev/admin-only until roles land.
- [Risk] `requireApproved` is forgotten at a future banking call site → Mitigation: make it the documented single entry point and cover it with integration tests for account creation and deposit/withdraw.
- [Risk] Sensitive identity data leaks via logs or responses → Mitigation: never log document numbers, storage keys, or identity payloads; restrict the status DTO to high-level fields.
- [Risk] CCCD/tax-id formats vary → Mitigation: enforce CCCD `^[0-9]{12}$` via DB check and Bean Validation; tax-id checks belong to the later KYB slice.
- [Risk] Status mapping duplication across auth/onboarding/banking codes → Mitigation: add `403/404/409` mappings now; move mapping onto the `ErrorCode` enum once all blocks exist, per exception.md.

## Migration Plan

1. Add enums, entities, and repositories; declare unique constraints, indexes, and the CCCD check via JPA annotations so Hibernate `ddl-auto` generates the onboarding tables. No Flyway in this slice.
2. Add error codes `2300`–`2308`; extend `GlobalExceptionHandler` for `403/404/409`.
3. Add DTOs and `OnboardingService` (profile, KYC, documents, submit, `requireApproved`).
4. Add the customer onboarding controller.
5. Add submit validation and audit logging.
6. Add `OnboardingReviewService` and reviewer controller behind the temporary admin gate.
7. Add `requireApproved` calls to account creation and money movement (when those services exist).
8. Add unit and integration tests; run `mvn test`.
9. Update `docs/design/api.md` and `docs/design/database.md`.

Rollback: remove the onboarding entities/endpoints/services. Since the schema is generated from entities under `ddl-auto`, removing the entities removes the tables on next start; no migration to revert. `users` rows remain valid; no auth behavior changes.
